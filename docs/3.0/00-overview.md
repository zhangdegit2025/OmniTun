# OmniTun 3.0 — 平台化总览

> 目标：在 2.0 满血核心之上，构建完整的商业化 SaaS 平台，实现从"能用的工具"到"可卖的 SaaS 产品"的跃迁。

## 一、为什么需要 3.0？

### 1.1 当前状态回顾

| 版本 | 交付物 | 状态 |
|------|--------|------|
| **Phase 1** (`docs/`) | 产品愿景、技术架构、安全模型、API 设计 | 长期蓝图层，持续迭代基准 |
| **2.0** (`docs/2.0/`) | 端到端隧道贯通、TLS+域名、企业安全(SSO/MFA/RBAC)、P2P+Mesh、可观测性、计费(Stripe) | 满血核心，可直接投产 |

### 1.2 满血 ≠ 可商业化

2.0 交付的是一个**技术完整的产品核心**——隧道能通、TLS 能签、用户能登录、钱能收。但从 SaaS 商业化视角审视，以下关键缺口仍然存在：

| 缺口 | 严重性 | 影响 |
|------|--------|------|
| **无运营管理后台** | 🔴 致命 | 平台方无法管理用户、组织、Relay 节点、滥用行为。所有操作依赖直接操作数据库。 |
| **用户端体验单薄** | 🟡 重度 | 当前 Dashboard 仅覆盖基础 CRUD，缺乏企业级用户预期的用量分析、团队协作、安全配置深度。 |
| **开发者生态空白** | 🟡 重度 | SDK 仅 Go/Python/JS 骨架，API 文档不完整，无开发者门户。平台无法形成网络效应。 |
| **企业级能力缺失** | 🟡 重度 | 私有部署、数据驻留、高级审计、SIEM 集成等在 2.0 中被明确排除。 |
| **商业运营能力零** | 🟡 重度 | 无 CRM、无销售流程、无客户成功体系、无数据分析驱动增长的基建。 |

### 1.3 竞品对标差距

| 能力维度 | ngrok | Cloudflare Tunnel | OmniTun 2.0 | OmniTun 3.0 |
|----------|-------|-------------------|-------------|-------------|
| 隧道穿透 | ✅ | ✅ | ✅ (满血) | ✅ |
| P2P / Mesh | ❌ | ❌ | ✅ (满血) | ✅ |
| 管理后台 | ✅ | ✅ | ❌ | ✅ |
| 开发者门户 | ✅ (API Docs) | ✅ | ❌ | ✅ |
| 多语言 SDK | ✅ (9种) | ✅ (多语言) | ⚠️ (骨架) | ✅ |
| 企业私有部署 | ✅ Enterprise | ✅ | ❌ | ✅ |
| 商业运营体系 | ✅ | ✅ | ❌ | ✅ |

结论：**2.0 让我们在核心技术维度追平甚至超越竞品，但在平台化、商业化维度仍然是零。**

### 1.4 2.0 满血核心能力清单

> 3.0 构建在 2.0 满血核心之上，**不重写、不替代**，仅在此基础上扩展商业层。以下列出 2.0 已完整交付的能力与 3.0 的增量。

| 能力域 | 2.0 满血交付 | 3.0 增量 |
|--------|-------------|----------|
| **隧道核心** | TCP/HTTP/HTTPS 端到端穿透，多协议支持 | —（核心引擎不变） |
| **TLS / 域名** | Let's Encrypt ACME 自动签发、自定义域名 | 管理后台证书监控与告警面板 |
| **认证体系** | OIDC SSO (Google/GitHub/AzureAD)、Email + MFA、组织级 RBAC 四角色 | Super Admin 独立认证体系 + 五级角色层级 |
| **企业安全** | RBAC (Owner/Admin/Editor/Viewer)、审计日志写入 | 安全事件中心、滥用检测规则引擎、SIEM 集成 |
| **P2P + Mesh** | P2P 直连传输、Mesh 组网、NAT 穿透 | 全球拓扑可视化、P2P 延迟矩阵实时图 |
| **可观测性** | Prometheus Metrics、结构化日志、基础 Dashboard | 管理后台全局仪表板、跨租户监控 |
| **计费体系** | Stripe 订阅管理、按量计费 | 多计划定价引擎、折扣/优惠券、税务自动化 |
| **用户端** | Dashboard 基础 CRUD（隧道/域名/API Key/设置） | Onboarding Wizard、通知中心、Cmd+K 全局搜索、深色模式 |
| **API** | REST API v1（核心 CRUD） | Admin API 命名空间、SDK(Go/Python/JS/TS) 完整发布 |
| **开发者工具** | CLI 基础命令 (login/connect/list)、SDK 骨架 | CLI 增强、交互式 API 文档、"Try It" 试玩、下载中心 |
| **运营管理** | ❌ 无 | 管理后台（组织/用户/Relay/安全/公告/Feature Flag/系统配置） |
| **企业级能力** | ❌ 无 | 私有部署 Helm Chart、SCIM、审计报告、SLA 追踪 |
| **商业运营** | ❌ 无 | 收入仪表板、客户 360、CRM 集成、客户成功体系 |

## 二、3.0 的五大支柱

一个完整的商业化 SaaS 平台由以下五根支柱构成，缺一不可：

```
                          ┌──────────────────────────┐
                          │      OmniTun 3.0          │
                          │    Complete SaaS Platform │
                          └──────────┬───────────────┘
                                     │
        ┌────────┬─────────┬─────────┼─────────┬──────────┐
        │        │         │         │         │          │
   ┌────▼──┐ ┌──▼───┐ ┌───▼───┐ ┌───▼───┐ ┌───▼───┐
   │ Admin │ │ User │ │ Dev   │ │Enter- │ │Business│
   │Console│ │Dash- │ │Plat-  │ │prise  │ │Ops     │
   │       │ │board │ │form   │ │Feat-  │ │        │
   │       │ │      │ │       │ │ures   │ │        │
   └───────┘ └──────┘ └───────┘ └───────┘ └───────┘
```

### 支柱一：管理后台（Admin Console）

**`admin.omnitun.io`** — OmniTun 内部运营人员的指挥中心。

| 能力域 | 核心功能 |
|--------|----------|
| 全局仪表板 | 平台健康指标、流量趋势、异常告警 |
| 组织管理 | 所有租户的增删改查、冻结、计划变更、模拟登录 |
| 用户管理 | 全局用户搜索、详情、会话管理、封禁 |
| Relay 节点管理 | 全球节点监控、拓扑视图、Drain/下线操作 |
| 证书管理 | 系统/租户 TLS 证书状态和操作 |
| 安全与滥用 | 滥用检测规则、举报队列、IP 黑名单、安全事件 |
| Feature Flag | 功能灰度发布、百分比/白名单策略 |
| 全局公告 | 公告创建、定时发布、Dashboard 横幅 |
| 系统配置 | 速率限制、邮件模板、维护模式、日志级别 |

> 详见：[01-admin-console.md](./01-admin-console.md)

### 支柱二：用户端（User Dashboard）

**`app.omnitun.io`** — 最终用户（组织内的 Owner/Admin/Editor/Viewer）的自助管理界面。2.0 已交付基础 CRUD，3.0 需补全企业级深度。

| 能力域 | 待增强内容 |
|--------|-----------|
| 仪表板增强 | 用量预测、异常检测通知、P2P 拓扑实时图 |
| 团队协作 | 团队空间(Workspace)完善、审批工作流、成员邀请链接 |
| 安全中心 | 安全评分、异常登录告警、审计日志增强搜索 |
| 开发者体验 | 隧道模板、快速部署向导、在线 CLI 生成器 |
| 计费中心 | 账单预测、预算告警、多币种支持 |

> 详见：[02-user-dashboard.md](./02-user-dashboard.md)

### 支柱三：开发者平台（Developer Platform）

**`developers.omnitun.io`** — 面向开发者的自助接入平台，是平台网络效应的引擎。

| 能力域 | 核心功能 |
|--------|----------|
| API 文档 | OpenAPI 3.1 交互式文档、代码示例、请求试玩 |
| SDK 中心 | Go/Python/JS/Java SDK 完善，源码+文档+示例项目 |
| CLI 增强 | 插件系统、守护进程模式、零配置模式 |
| Webhook | 隧道事件 Webhook 管理、重试策略、投递日志 |
| 集成市场 | 主流 DevOps 工具集成（GitHub Actions, Vercel, K8s） |
| 变更日志 | API 版本演进、Breaking Change 通知 |

> 详见：[03-developer-platform.md](./03-developer-platform.md)

### 支柱四：企业级能力（Enterprise Features）

面向付费企业客户的核心竞争力——不是开源项目能快速复制的部分。

| 能力域 | 核心功能 |
|--------|----------|
| 私有部署 | Helm Chart 完整版、Air-gapped 部署、自托管升级 |
| 数据驻留 | 按大洲/国家选择数据处理区域 |
| 高级安全 | Mutual TLS、OPA 策略引擎、SIEM 集成(Splunk/Elastic/Sentinel) |
| 合规 | SOC2 Type II 准备、GDPR 数据处理协议、审计报告导出 |
| SLA | 99.9% / 99.99% 分级 SLA、专属客户成功经理 |
| 专用基础设施 | 企业专属 Relay 节点、专用 IP、BYOK 加密 |

> 详见：[04-enterprise-features.md](./04-enterprise-features.md)

### 支柱五：商业运营（Business Operations）

让平台持续增长并盈利的运营系统——Sales、Marketing、Customer Success、Analytics 的数字化基建。

| 能力域 | 核心功能 |
|--------|----------|
| 产品分析 | 漏斗分析（注册→激活→付费）、留存分析、功能使用率 |
| CRM 集成 | HubSpot/Salesforce 集成、客户 360 视图、合同管理 |
| 客户成功 | 健康评分、流失预警、自动触达、Onboarding 进度追踪 |
| 营销基建 | 官网 CMS、Blog/Changelog、邮件营销、SEO |
| 财务运营 | 多计划定价引擎、折扣/优惠券、多币种、税务自动化 |
| 合规运营 | SLA 合约管理、DPA 签署、审计报告交付 |

> 详见：[05-business-operations.md](./05-business-operations.md)

## 四、与 Phase 1 和 2.0 的关系

```
┌─────────────────────────────────────────────────────────────┐
│                     Phase 1 (docs/)                          │
│         产品愿景 · 技术架构 · 安全模型 · API 设计            │
│                   长期蓝图层，持续迭代                        │
└────────────────────────┬────────────────────────────────────┘
                         │
         ┌───────────────┴───────────────┐
         │                               │
┌────────▼──────────┐          ┌─────────▼─────────┐
│  2.0 (docs/2.0/)  │          │  3.0 (docs/3.0/)  │
│   满血核心引擎     │  ───►    │   平台化商业层     │
│                   │  依托于   │                   │
│ · 隧道贯通        │          │ · 管理后台        │
│ · TLS + 域名      │          │ · 用户端增强      │
│ · 企业安全        │          │ · 开发者平台      │
│ · P2P + Mesh      │          │ · 企业级能力      │
│ · 可观测性        │          │ · 商业运营        │
│ · 计费上线        │          │                   │
└───────────────────┘          └───────────────────┘
```

**分层逻辑**：
- **2.0 = 核心引擎**：定义"产品能做什么"——隧道能通、TLS 能签、用户能管。
- **3.0 = 平台化层**：定义"平台如何运作"——谁来管理平台、用户如何自助、开发者如何接入、企业如何采购、业务如何增长。
- 3.0 构建在 2.0 之上，**不修改 2.0 核心架构**，而是扩展新的服务、新的前端应用、新的运营流程。
- Phase 1 文档提供**所有层的设计原则和约束**，2.0 和 3.0 各取其子集进行深度实现。

## 五、实施优先级

### Phase 3.0-A：运营基础（P0，必须）

> **先让平台可运营，再谈增长。**

1. **管理后台**（01-admin-console.md 完整实施）
   - 没有管理后台，平台就是盲飞状态
   - 组织管理、用户管理、Relay 监控、安全与滥用管理
2. **Super Admin 角色与权限**
   - 支撑管理后台的认证/授权基础设施

### Phase 3.0-B：商业闭环（P1，高优）

> **平台能赚钱、能留住客户。**

3. **企业级能力**（04-enterprise-features.md 核心项）
   - 私有部署、数据驻留、SLA 框架
4. **商业运营基建**（05-business-operations.md 核心项）
   - 定价引擎、产品分析、客户成功体系

### Phase 3.0-C：生态发力（P2，重要）

> **网络效应开始形成。**

5. **用户端增强**（02-user-dashboard.md）
6. **开发者平台**（03-developer-platform.md）
