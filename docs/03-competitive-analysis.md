## 二、竞品逐个剖析

### 2.1 ngrok

| 维度 | 评估 |
|------|------|
| **成立** | 2015 年，YC W15，总融资 $70M+ |
| **市场地位** | 品类代名词，品牌认知度全球第一 |
| **产品能力** | ★★★★☆ |
| **定价** | ★★☆☆☆（被开发者长期诟病"太贵"） |
| **开发者体验** | ★★★★★（行业标杆） |

**核心优势**：
- 品牌即品类：开发者说"ngrok 一下"如"Google 一下"
- CLI 体验极致：Agent 配置秒级同步，断线重连无感
- Ingress 生态：K8s Ingress Controller + Docker 集成
- 边缘函数：可在隧道上运行 Lua/JS 脚本修改请求

**致命缺陷**：
1. **定价激进**：UDP 隧道仅 Enterprise（$65/人/月），P2P 直连不提供。大量开发者用 frp 只为"付不起 ngrok"
2. **封闭生态**：服务端不开源，数据经 ngrok 服务器，企业合规存疑
3. **无 Mesh 组网**：不能做私有网络，不能替代 VPN
4. **架构老化**：WebSocket 长连接不够高效，QUIC 支持滞后

**OmniTun 对策**：
- 直接对标：`omnitun http 8080` 体验一致，但 Pro 即开放 UDP + P2P
- 价格切入：$12 vs $20（Pro vs ngrok Pro），差距 40%
- 协议覆盖：ngrok 的协议白名单是弱点，我们全协议覆盖

### 2.2 Cloudflare Tunnel（原 Argo Tunnel）

| 维度 | 评估 |
|------|------|
| **成立** | Cloudflare 旗下 |
| **市场地位** | 企业级首选（零信任） |
| **产品能力** | ★★★★★ |
| **定价** | ★★★★☆（Free 层慷慨） |
| **开发者体验** | ★★★☆☆（cloudflared CLI 尚可但配置复杂） |

**核心优势**：
- 零信任安全模型完整：WARP Client + Access + Gateway + Tunnel 形成闭环
- 与 Cloudflare CDN 深度集成：自动 DDoS 防护、全球边缘缓存
- 免费层慷慨：无限隧道、自定义域名、DDoS 防护
- 企业生态：与 Access（SSO）、Gateway（DNS 过滤）、WAF 无缝结合

**致命缺陷**：
1. **生态锁定**：隧道终结于 Cloudflare 边缘，流量必须经 Cloudflare 网络。无法脱离 CF 使用
2. **协议受限**：UDP 支持非常有限，ICMP 不支持，非 HTTP 协议体验差
3. **国内问题**：Cloudflare 在中国无直接 POP，延迟高且不稳定
4. **非开发工具思维**：cloudflared 是运维工具，不是开发者工具。没有请求检查、重放、Mock
5. **无私有组网**：不能替代 Tailscale/ZeroTier

**OmniTun 对策**：
- 我们不是 CDN 公司，不绑生态，保持协议中立
- 聚焦"开发者工具"而非"企业安全产品"，体验差距是核心竞争力
- 国内部署独立节点，解决延迟问题

**定价备注**：Cloudflare Tunnel 本身免费，但 SSO/OIDC 功能需要 Cloudflare Access（$4/用户/月 起），WAF 等企业功能需要 Enterprise 套餐。

### 2.3 Tailscale

| 维度 | 评估 |
|------|------|
| **成立** | 2019 年，总融资 $180M+ |
| **市场地位** | Mesh VPN 品类第一 |
| **产品能力** | ★★★★★（Mesh 极强） |
| **定价** | ★★★★☆ |
| **开发者体验** | ★★★★★ |

**核心优势**：
- WireGuard 内核 + 自研 DERP 中继 + NAT 穿透引擎，P2P 连接成功率极高
- Control 面设计优雅：基于 OIDC / SSO，ACL 用 policy 文件声明
- 客户端体验最佳：安装即用，零配置
- 开源客户端（tailscale/tailscale），社区信任度高

**致命缺陷**：
1. **没有公网暴露能力**：Funnel 功能只在 beta 且限制极大（仅 HTTP、有带宽限制）
2. **服务端不开源**：协调服务器（coordination server）闭源，企业私有部署受限
3. **不支持 TCP/UDP 隧道到公网**：无法替代 ngrok
4. **不适合 IoT 大规模部署**：每个设备一个 node，管理成本高

**OmniTun 对策**：
- 把 Tailscale 的 Mesh 能力内置，但加上公网暴露能力 = 一个产品解决两个问题
- 面向隧道场景优化，而非"替代 VPN"
- IoT 设备管理：设备分组 + 批量配置 + 远程命令

### 2.4 frp（Fast Reverse Proxy）

| 维度 | 评估 |
|------|------|
| **成立** | 2017 年，GitHub 87k+ Stars |
| **市场地位** | 最大开源内网穿透工具 |
| **产品能力** | ★★★☆☆（功能全但体验粗糙） |
| **定价** | ★★★★★（免费） |
| **开发者体验** | ★★☆☆☆ |

**核心优势**：
- GitHub 星数断层领先，社区生态强
- 功能覆盖全面：TCP/UDP/HTTP/HTTPS/STCP/XTCP/P2P
- 配置灵活：支持多种协议、负载均衡、加密压缩
- 轻量高性能：Go 编写，内存 <10MB

**致命缺陷**：
1. **无 SaaS 多租户**：一台服务器一个 frps 实例，手动管理用户
2. **无 Web Dashboard**：控制面全靠配置文件
3. **安全模型薄弱**：无 RBAC、无审计、无 SSO、密钥分发靠手动
4. **运维门槛极高**：需自建服务器、配置 TLS、管理 DNS、监控告警
5. **P2P 成功率低**：XTCP 打洞依赖 frps 辅助，无 STUN/TURN

**OmniTun 对策**：
- frp 是我们的"用户蓄水池"——frp 用户痛苦升级就是 OmniTun 的目标用户
- 开源核心引擎对标 frp 的能力，但加上 SaaS 控制面和 Web Dashboard
- 提供 frp 迁移工具（`omnitun migrate frp`）

### 2.5 花生壳（Oray）

| 维度 | 评估 |
|------|------|
| **成立** | 2004 年，国内最老牌 |
| **市场地位** | 国内中小企业首选 |
| **产品能力** | ★★☆☆☆ |
| **定价** | ★★☆☆☆（国内偏高） |
| **开发者体验** | ★☆☆☆☆ |

**核心优势**：
- 国内品牌认知度最高，运营 20 年
- 多级代理资质，合规路径清晰
- 硬件盒子嵌入式方案成熟

**致命缺陷**：
1. 技术栈老旧，性能差，限速严重
2. 不面向现代开发者（无 CLI 体验、无 API、无 SDK、无 CI/CD 集成）
3. 功能单一：难支持 UDP、WS、gRPC、P2P
4. 封闭，无法 DevOps/Infra as Code 管理

**OmniTun 对策**：
- 花生壳代表"上一代"，OmniTun 代表"下一代"
- 国内合规路径参考但技术全面超越
- 提供迁移工具降低切换摩擦

## 四、护城河分析

| 护城河类型 | OmniTun 的能力 | 可持续性 |
|------------|---------------|----------|
| **技术壁垒** | 自适应拓扑引擎 + 全协议统一向量流 | 中（2-3年代差） |
| **网络效应** | Relay 节点越多延迟越低，用户越多网络越强 | 高 |
| **数据飞轮** | NAT 类型数据提升打洞成功率 → 更多用户贡献数据 | 高 |
| **生态锁定** | IDE 插件 + CI/CD 集成 + SDK | 中高 |
| **品牌认知** | "ngrok 替代品"的心智抢占 | 中 |
| **合规壁垒** | 国内 ICP/等保资质 | 高（时间+资金门槛） |
